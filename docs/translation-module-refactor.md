# 翻译模块上下文化改造说明

## 目标与可行性

本次改造保留现有模型接入、翻译队列和 `translation_content` 展示结构，只重构送模前处理、术语注入、质量校验及人工反馈闭环，因此不需要更换模型，也不影响日报原文字段。

新管线默认启用，旧占位符管线保留为运行时回滚选项。翻译版本包含管线、模型、Prompt、业务模板、术语及保护项指纹，配置变化后旧译文会自动进入待重译状态。

## 新的数据流

1. 保存 PDF 解析后的原始字段，不覆盖原文。
2. 仅清除高置信度 PDF 页眉/页脚和跨行断词，同时记录清洗前后文本。
3. 以完整作业事件字段为翻译单元，注入本行时间、作业代码、井段和相邻动作。
4. 只匹配当前原文实际出现的术语，按严格保护词、固定标准术语、上下文术语、行业短语传给模型。
5. 上下文管线把完整清洗后原文发送给模型；旧版管线才使用占位符。
6. 首轮译文完成后由程序检查数字、编号、单位、动作和标准术语。
7. 数字、严格保护项或单位校验失败时，携带初译和具体错误做定向修正；不做无条件多轮润色。
8. 保存原文、实际送模请求、模型原始响应、终译、校验结果和耗时。
9. 人工修订保留修改前后版本，可选择沉淀为跨模型版本复用的确认译例。

## 配置与界面

- “字段与 Prompt”：选择上下文管线或旧版回滚管线；分别维护钻井、完井、修井和搬迁业务提示。
- “术语词库”：维护术语类型、优先级及是否严格原样保留；Excel 导入导出支持相同字段。
- “测试工作台”：查看实际送模原文、命中术语、严格保护项、完整 Prompt 和质量检查；支持新旧管线同文对比。
- “翻译记忆与日志”：维护人工确认完整译例，按日报逐条修订译文，并检查清洗、请求、原始响应和终译链路。

## 数据库迁移

`db/init.sql` 新增：

- `translation_memory`：人工确认的完整句/段译例，不受当前 Prompt 版本限制；
- `translation_revision`：人工修订前后文本、操作人、说明和时间。

应用启动时继续执行幂等的 `CREATE TABLE IF NOT EXISTS`。现有 `translation_content` 不迁移、不删除，仍用于日报译文和同版本精确缓存。

## 发布顺序

1. 备份 MySQL 与 `outputs/system_config.json`、翻译配置、术语配置。
2. 发布代码并启动一次服务，让幂等数据库初始化创建新表。
3. 在测试工作台用真实日报执行单条测试和新旧管线对比。
4. 先选择少量日报重译，核对完整调试链路和人工修订流程。
5. 扩大到全部待处理日报；旧译文不会在配置变更时被静默复用。
6. 如出现不可接受的质量回退，在“翻译管线”选择旧版占位符并保存，再对受影响日报重译。

## 验证标准

- 原始数据库字段不被预处理或译文覆盖；
- 上下文管线实际送模文本不含 `[[P0]]` 等占位符；
- 只注入当前文本命中的术语；
- 数字集合、BHA/井号/型号及单位语义通过确定性检查；
- 相邻作业只作为 `event_context`，不出现在译文中；
- 校验失败时只重试失败项，并携带初译与错误原因；
- 人工修改可查询到修订记录，勾选后可在后续相同原文中命中翻译记忆；
- 调试日志不包含 API Key 或 Authorization 请求头。

## 后续测试集

生产验收应从真实日报中抽取 50～200 个已人工确认事件，覆盖钻井、循环、起下钻、BHA、固井、完井、修井、NPT、泥浆、HSSE 和搬迁。每次调整模型、Prompt 或词库后，固定比较新旧版本的数字完整率、保护项完整率、术语一致率、动作覆盖率和人工接受率。
